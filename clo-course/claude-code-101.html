Part 1

What Is Claude Code?

An AI tool that reads your project, edits files, runs commands, and integrates with your development tools. Not a chatbot -- an AI employee.

New to the terminal or Git? Start with the "Start Here" sections at the top of the sidebar.

Regular AI Chatbot

  • You type, it responds
  • No access to your files
  • No memory between chats
  • One response at a time
  • You copy-paste output
  • General knowledge only

Claude Code

  • Reads your files, edits them, runs commands
  • Reads your ENTIRE project
  • Remembers YOUR project via CLAUDE.md
  • Works through multi-step tasks autonomously
  • Writes directly into your files
  • Knows YOUR project, YOUR processes, YOUR tools
What you can tell Claude Code to do
"Write tests for the auth module, run them, and fix any failures"
"Commit my changes with a descriptive message"
"Review PR #456 and suggest improvements"
"What are the most common errors in the last 24 hours?"
"Translate new strings into French and raise a PR for review"
Extend with 5 building blocks
CLAUDE.md (Memory) → Skills (Capabilities) → Sub-Agents (Delegation) → MCP (External Tools) → Hooks (Automation)

Each building block adds a layer. You'll learn them all in this course, progressively.

Part 1

Where Claude Code Lives

Same engine, 6 environments. Your CLAUDE.md, settings, and MCP servers work across all of them.

Why Claude Code?

Most AI tools give you a chatbot. Claude Code gives you an AI employee you can train. You write a CLAUDE.md and it learns your project. You create skills and it learns your processes. You connect MCP servers and it can reach your tools. You set permissions and it knows its boundaries. No other AI coding tool gives you this level of customisation — skills, memory, sub-agents, hooks, and MCP connections all working together. The result? An AI that doesn't just write code. It writes code your way, follows your standards, and gets better every time you use it.

Your Workspace

A workspace is just a folder on your computer that you open in VS Code. When you add a .claude/ folder with instructions inside, Claude reads those instructions and knows what to do. Without .claude/ you have a generic chatbot. With it, you have an AI employee that follows your processes.

Part 1

How Claude Code Thinks

Not autocomplete. Claude Code plans, executes, verifies, and adjusts -- like a new hire who reads the manual and checks their own work.

The Think-Act-Check Loop

Claude works in a loop: gather context, act, verify, repeat until done. (Sometimes called an "agentic loop" — just means it works in a cycle.)

This is NOT autocomplete. Claude Code doesn't suggest the next line -- it works through entire tasks end-to-end. Think: "A new hire who reads the manual, tries the task, checks their own work, and adjusts."
Your Conversations Are Saved

Every conversation with Claude is automatically saved. You can start a new one anytime or pick up an old one from your chat history in VS Code.

Continue a conversation
Open the chat history panel and select a previous conversation. Claude remembers everything from that session.
Start fresh
Start a new conversation when switching tasks. Keeps context clean and Claude focused on the new task.

Your old conversations are never lost -- you can always go back and review them.

What About Rename, Delete, Move?

There are no dedicated tools for these. Claude uses Bash to run shell commands for anything not covered by the 8 tools above:

Rename a file → mv old.ts new.ts
Delete a file → rm file.ts
Move a file → mv file.ts src/
Install packages → npm install axios

This is why Bash appears in both Take action and Verify — it's the Swiss Army knife. Claude will ask permission before running anything destructive.

Plus MCP tools: When you connect external services (Notion, GitHub, Slack), Claude gets additional tools from those servers. More on this in the MCP section.

The most important file in your workspace. First thing Claude reads every session. If a skill isn't mentioned here, Claude won't know it exists.

What CLAUDE.md Looks Like

Think of CLAUDE.md as your project's instruction manual. It tells Claude: what skills to load, where files go, and what rules to follow.

Why CLAUDE.md Matters
What Goes in CLAUDE.md

Three sections that make Claude useful from the first message:

1. Skill Routing
Tell Claude which skill to load for each type of request.
User wants to write content → write User wants to create visuals → visual-content User wants to analyze business → business-x-ray
2. Where Things Live
Tell Claude your project structure so it puts files in the right place.
.claude/skills/ → Skills clients/ → Client work content/ → Output
3. Rules & Conventions
Tell Claude how you work -- standards it should always follow.
Use markdown for content Run tests before committing Follow brand voice guide
Where CLAUDE.md Lives

Imports: Pull In Other Files

Use @path/to/file in your CLAUDE.md to pull in content from other files. Think of it like linking -- Claude reads the linked file automatically.

@README → Pulls in your project readme so Claude understands the project
@docs/architecture.md → Pulls in architecture docs so Claude follows your structure
@.claude/rules/security.md → Pulls in security rules Claude must always follow
Imports chain: Imported files can import other files too (up to 5 levels deep). This lets you keep CLAUDE.md short while still giving Claude access to everything it needs.
Quick start: Run /init to auto-generate a CLAUDE.md for your project. Run /memory to edit it anytime.

The Memory Hierarchy

6 layers of memory, from org-wide policies down to automatic learnings. More specific always wins. Click each layer to explore.

For most VS Code users, you'll use layers 2-6. Layer 1 (Managed Policy) is for IT admins at large organizations. If you're a solo user or small team, start with Project Memory (your CLAUDE.md) and Auto Memory (happens automatically).
1. Managed Policy -- Org-wide rules (enterprise only)
2. Project Memory -- Your CLAUDE.md (shared with team)
3. Project Rules -- .claude/rules/ (topic files)
4. User Memory -- Your personal preferences (all projects)
5. Local Memory -- Personal, this project only
6. Auto Memory -- Claude learns as you work
Click a layer above to see details.
Modular Rules

Instead of one giant CLAUDE.md, organize instructions into topic files:

.claude/rules/ code-style.md -- Code style guidelines testing.md -- Testing conventions security.md -- Security requirements frontend/ react.md -- React-specific rules styles.md -- Styling conventions

All .md files in .claude/rules/ are auto-loaded as project memory. Path-scoped rules can target specific file patterns with YAML frontmatter.

Claude also learns over time. As you work, Claude automatically remembers project patterns, build commands, and your preferences. This happens in the background — you don't need to manage it. You can also tell Claude directly: "remember that we always use dark mode for demos."

Part 4: Skills

What Is a Skill?

Skills extend what Claude can do. Create a SKILL.md file with instructions, and Claude adds it to its toolkit. This is how you build AI Employees.

A Skill Is Just a Text File

Think of a skill like a recipe card. It tells Claude: what this skill is for, when to use it, and how to do the work. The settings at the top use YAML — a simple format where you just write key: value pairs.

Two Ways Skills Get Used

Auto-loaded

  • Your request matches the skill description
  • Claude loads it automatically
  • You don't need to do anything special
# You say: "How does this code work?" # Claude auto-loads explain-code skill

Direct invoke

  • You call it with /skill-name
  • Loads immediately with arguments
  • Good for specific, intentional actions
# You type: /explain-code src/auth/login.ts # Skill loads with that file as argument

How Skills Get Triggered

Three modes, arguments, and the refinement loop. Everything you need to know about using skills day-to-day.

Three Invocation Modes

Arguments: Filling in the Blanks

When you type /skill-name followed by extra words, those words become the arguments. Think of it like a form with a blank field — $ARGUMENTS is where your words get filled in.

Without arguments

# You type: /follow-up

Claude has to ask: "Who is the follow-up for?"

With arguments

# You type: /follow-up John about the proposal

Claude starts immediately — it already knows the context.

How it works inside your SKILL.md:
Write a follow-up email for $ARGUMENTS
When you type /follow-up John about the proposal, Claude sees:
"Write a follow-up email for John about the proposal"
Use $ARGUMENTS when...
  • You invoke skills with /name
  • The skill needs specific input each time
  • Example: /write thread about AI tools
Skip $ARGUMENTS when...
  • Claude auto-loads the skill from your message
  • The skill always does the same thing
  • Example: skill that checks code quality
Don't worry about forgetting $ARGUMENTS. If your skill doesn't include $ARGUMENTS anywhere, Claude automatically appends whatever you typed after the slash command. It just works.
The Refinement Loop

Most skills need 3-5 rounds of refinement. That's normal -- you're training an employee.

Build Your First Skill

A step-by-step walkthrough. By the end, you'll have a working skill that writes client follow-up emails in your voice.

The Goal

Instead of typing "write me a follow-up email" and getting generic AI output every time, you'll type /follow-up and get an email that sounds like you, follows your format, and matches your tone. We'll build this in 3 stages.

Stage 1: Basic
Stage 2: + Voice
Stage 3: + Workflows
Stage 1: Just Instructions

Create one file. That's it.

Create this file:
📄 .claude/skills/follow-up/SKILL.md
---
name: follow-up
description: Write client follow-up emails
---
Write a follow-up email for $ARGUMENTS.

Keep it professional but warm.
Include a clear next step.
Keep it under 150 words.
📁 .claude/skills/follow-up/ └── 📄 SKILL.md
You type: /follow-up post-call with Sarah
Claude's output:
Hi Sarah,

Great speaking with you today. I wanted to follow up on our conversation about...

(Generic but functional. It works, but it doesn't sound like YOU.)
Problem: Sounds like any AI. No personality, no specific format.
Stage 2: Add Your Voice

Add one reference file with examples of how you actually write.

Add this file:
📄 .claude/skills/follow-up/references/my-voice.md
My writing style:
- I start emails with the person's name, no "Hi"
- I keep sentences short. Punchy.
- I always end with a specific date/time
- I use "let's" not "I would like to"

Example email I've sent:
Sarah,

Great call. Two things stood out...

Let's lock in next Tuesday at 2pm
to walk through the proposal.

Talk soon.
📁 .claude/skills/follow-up/ ├── 📄 SKILL.md └── 📁 references/ └── 📄 my-voice.md
Same prompt: /follow-up post-call with Sarah
Claude's output (with your voice):
Sarah,

Great call. The two things that stood out: your onboarding bottleneck and the reporting gap.

Let's lock in Thursday at 3pm to map out a solution. I'll send a short brief beforehand.

Talk soon.
Same command, completely different output. Now it sounds like you.

Different follow-up types = different workflows. One command, multiple outputs.

Add workflow files:

One skill, three outputs — same voice:

Skill Deep Dive

Now that you've built one, here's the full toolkit — where skills can live, what all the settings do, and how to pass data.

Skills can live in 4 places. Where you put them determines who can use them:

What Are Enterprise Skills?
Enterprise skills are deployed by IT admins via managed settings -- users can't edit or remove them. They enforce organization-wide standards (coding conventions, security policies, deployment procedures) across every user's Claude Code. Most small teams won't need these -- project-level skills committed to git achieve the same "everyone gets the same skill" effect.
Two Types of Skill Content

Reference Content

  • Knowledge Claude applies passively
  • Conventions, patterns, style guides
  • Runs inline (no special invocation)

Task Content

  • Step-by-step instructions
  • Specific actions with defined output
  • Often invoked with /skill-name

Your CLAUDE.md is the Traffic Controller

The routing table in CLAUDE.md is the single most important part of your workspace. If routing is wrong, the wrong skill loads, and the output is wrong. Here's how to structure it.

📄 .claude/CLAUDE.md -- Routing Table
## Skill Routing
| User wants to... | Load skill |
|-----------------|------------|
| Write emails, content | write |
| Generate client reports | reporting |
| Create visuals, diagrams | visual-content |
| Manage tasks, daily plan | war-map |
Why this matters: Without a routing table, Claude guesses. With one, Claude knows exactly which skill to load. If you have 10 skills, Claude reads this table and picks the right one in milliseconds. No confusion, no wrong outputs.
When Routing Goes Wrong (and How to Fix It)
Problem: Wrong skill loads
You say "create a report" but the write skill loads instead of reporting.

Fix: Make routing table entries more specific. Change "Write anything" to "Write emails, newsletters, social posts" so it doesn't catch "report".
Problem: Generic output for specific client
Your reporting skill creates a generic report instead of using Acme Corp's format.

Fix: Add a client reference file. Put Acme's specifics (metrics, format, database ID) in references/acme-corp.md. Tell the SKILL.md to load it when Acme is mentioned.
Problem: Client workflows are too different
Acme needs Excel exports, Globex needs a dashboard. Same skill, totally different processes.

Fix: Use Pattern B (client folders). Create workflows/acme-corp/ and workflows/globex-inc/ with separate workflow files for each.
Problem: Skill is getting too complex
Your reporting skill is 500 lines and handles 8 clients with 5 workflow types each.

Fix: Split into separate skills per domain. reporting-weekly, reporting-reviews, reporting-dashboards. Update your CLAUDE.md routing table to match.
Want to automatically validate outputs? Hooks can check voice, accuracy, and formatting after every skill run. We'll cover this in the Hooks section.
You don't have to build this manually. Your workspace has a /create-skill command. Describe what you need — "I need a reporting skill that handles 3 different clients with different metrics" — and Claude will scaffold the folder structure, SKILL.md, workflows, and references for you.

How Fork Works

When a skill has context: fork, Claude branches off a separate conversation for the task. Your main conversation stays clean — only the summary comes back.

Without fork (default)
Everything runs in your main conversation. All files read, all output — stays in your context window. Good for quick tasks.
With context: fork
Branches off a separate context. The sub-agent reads files, drafts, revises — all in its own space. Only the clean summary comes back.
Sub-Agents + Skills = Specialists

Give a sub-agent a skill in its frontmatter and the full skill loads into the sub-agent's context — it becomes a specialist.

When to use fork: Skills that read lots of files (research, analysis) or produce verbose output (writing, reports). The sub-agent does the heavy work and sends back a clean summary.
Two Ways to Combine Them
Skill runs as Sub-Agent
Add context: fork to your skill. The skill content becomes the task.
--- name: write context: fork agent: general-purpose ---
Sub-Agent loads Skills
Add skills: to your agent. The full skill content is injected at startup.
--- name: api-developer skills: - api-conventions - error-handling ---
Key difference: With context: fork, the skill controls the task. With skills:, the sub-agent controls the task and uses skills as reference knowledge. Sub-agents don't inherit skills from the parent — you must list them explicitly.

What a Hook Looks Like

A JSON entry in your settings file. Maps directly to WHEN → WHAT → DO above. Three hook types: command (run a script), prompt (ask Claude a yes/no), agent (spawn a verifier).

Hook Lifecycle (click events to explore)
Session Start
Load memory, skills, MCP tools
SessionStart fires when Claude begins a new session. Use it to set up environment, check prerequisites, or load dynamic context. Cannot block.
You Submit Prompt
Your message enters the system
UserPromptSubmit fires when you send a message. Can block -- reject prompts that don't meet criteria. Use for input validation or routing.
Before Tool Use
Claude is about to use a tool
PreToolUse fires before any tool runs. Can block -- deny dangerous operations. Match on tool name: Write, Bash, Edit. Example: block writes to .env files.
Permission Check
Claude needs permission for something
PermissionRequest fires at permission dialogs. Can auto-allow or auto-deny. Use to auto-approve safe operations or block risky ones without user interaction.
Tool Executes
The tool runs and produces results
PostToolUse fires after a tool succeeds. Cannot block, but can give feedback to Claude. Example: after Write, check file for AI-sounding phrases and tell Claude to fix them.
Claude Finishes
Claude says "I'm done"
Stop fires when Claude finishes a response. Can force continue -- make Claude keep working if quality checks fail. This is the quality gate for output validation.
Session End
Session terminates
SessionEnd fires when the session ends. Cannot block. Use for cleanup, saving session summaries, or logging.
